# cgpt-bridge — Chrome Extension (Stage 2 MVP)

Stage 2 of the `cgpt-bridge` project. This stage proves that a Chrome MV3 extension can interact with the active ChatGPT tab end-to-end: insert a prompt, submit it, wait for the assistant response, and return the text to the background service worker.

This stage does **not** include the Rust CLI, the Native Messaging host, the agent protocol, or command execution. Those come in later milestones (see `../docs/roadmap.md`).

## Layout

```
extension/
  manifest.json
  package.json
  tsconfig.json
  vite.config.ts
  scripts/
    copy-manifest.mjs
  src/
    background.ts       # service worker; handles action click
    content.ts          # routes messages, calls the adapter
    chatgptAdapter.ts   # ALL ChatGPT-specific DOM logic lives here
    bridgeProtocol.ts   # internal extension message types
    errors.ts           # typed error classes + serialization helper
  dist/                 # built output (generated by `npm run build`)
```

The principle from `docs/architecture.md` applies: ChatGPT-specific selectors and DOM behavior are confined to `chatgptAdapter.ts`. UI breaks should be a one-file fix.

## Install dependencies

Requires Node.js 18+ and npm.

```sh
cd extension
npm install
```

## Build

```sh
npm run build
```

The build runs Vite twice (once for the background service worker, once for the content script) and copies `manifest.json` into `dist/`. The output is a self-contained unpacked extension at `extension/dist/`.

Other scripts:

- `npm run typecheck` — strict TypeScript check, no emit.
- `npm run clean` — remove `dist/`.

## Load the unpacked extension in Chrome

1. `cd extension && npm install && npm run build`
2. Open Chrome and go to `chrome://extensions`.
3. Enable **Developer mode** (toggle in the top-right).
4. Click **Load unpacked**.
5. Select `extension/dist/`.
6. Note the extension id Chrome assigns. You will need it in later milestones (M4 Native Messaging) for `allowed_origins`.

## Manual test

1. Open `https://chatgpt.com/` in Chrome and log in.
2. **Reload the ChatGPT tab** after first installing or reloading the extension. Chrome does not auto-inject content scripts into pages that were already open before the extension loaded.
3. Click the `cgpt-bridge` toolbar icon.
4. Watch the badge:
   - `...` while the test prompt is in flight.
   - `OK` on success.
   - `ERR` on failure.
5. Inspect the service-worker console (see below) for the assistant response text.

The hardcoded test prompt is `Say "pong" in one short sentence.` (defined in `src/background.ts`).

## Inspect logs

### Background service worker console

1. `chrome://extensions` → find `cgpt-bridge`.
2. Click the **service worker** link (visible while the worker is alive; it may say `Inactive` — click to wake it).
3. A DevTools window opens. Use the **Console** tab.

Look for lines prefixed with `[cgpt-bridge]`.

### Content script console

1. Open `https://chatgpt.com/` and right-click → **Inspect** to open the page DevTools.
2. The content script's `console.*` output appears in the page's regular Console tab.

The current content script does not log on its own; errors are surfaced back to the background worker via the response message.

## Known limitations

- **ChatGPT DOM can change without notice.** The adapter uses semantic selectors (ARIA roles, `data-testid`, `contenteditable` containers) and known stable structures like `[data-message-author-role="assistant"]`, but a UI refresh on ChatGPT's side may still break things. Repair in `src/chatgptAdapter.ts`; do not spread selectors elsewhere.
- **No Rust, no Native Messaging, no CLI.** This stage is browser-only. Triggering an `ask` requires a toolbar click.
- **No agent protocol.** No `cgpt-agent-response-v1` parsing, no plan storage, no command execution.
- **Hardcoded test prompt.** Defined in `src/background.ts`. Editing the prompt requires a rebuild.
- **Result is logged to console only.** Nothing is written to disk; nothing is sent anywhere off-machine.
- **Existing tabs must be reloaded** after installing/reloading the extension before the content script is injected.
- **Single active-tab assumption.** The action only targets the currently active tab in the focused window.
- **No clipboard, ever.** The composer is populated by direct DOM updates and dispatched input events.
